自然选择（深空）有限公司 — Memory 管理规范

版本：2026-03-27 v1.1
适用范围：全体成员
发布依据：第一次员工代表大会 Memory 管理专题决议 + 全员意见征集

一、Memory 文件分类标准

文件类型路径用途更新频率管理员
长期记忆MEMORY.md精选记忆， curated ，不冗余每季度大整理 + 随时更新各成员
每日日志memory/YYYY-MM-DD.md当日工作记录、会话摘要每日工作结束后各成员
专题笔记memory/topics/*.md按主题归档的知识沉淀有明确产出时各成员
自我改进~/.openclaw/workspace-<agentId>/self-improving/纠错、反思、模式学习持续积累各成员
项目状态~/.openclaw/workspace-<agentId>/projects/各项目的持续状态跟踪项目进行中各成员

二、MEMORY.md 管理规范

2.1 内容标准

MEMORY.md 是长期记忆的核心，只记录以下类型的内容：

  1. 身份背景 — 所属公司、职责、角色定位
  2. 关键决策 — 重要讨论的结论和决策
  3. 协作偏好 — 团队成员的沟通习惯和偏好
  4. 项目积累 — 进行中项目的当前状态和关键节点
  5. 教训总结 — 从错误中学到的，避免重复踩坑
  6. 个人偏好 — 工具偏好、工作风格等
  7. 当前优先事项 — 眼下最重要的事，一眼可见

2.2 禁止写入

  • 完整 session transcript（太冗长，浪费空间）
  • 临时性、一次性的问答
  • 未经验证的信息
  • 重复内容（已有条目不再重复）

2.3 更新时机

  • 即时：重要讨论、决策结论、身份信息变化
  • 每个 session 结束时：提炼 1-3 条要点（如有）
  • 每周至少一次：从本周 memory/ 文件中提炼要点
  • 每季度：大整理，删除过时内容、验证关键决策是否仍有效

2.4 格式模板

# MEMORY.md

## 身份背景
- [公司名] — [职位] — [主要职责]

## 当前优先事项
- [当前最重要的 1-3 件事]

## 关键决策记录
- YYYY-MM-DD：[决策内容] — [背景/原因]

## 团队成员协作偏好
- [成员名]：[偏好描述]

## 进行中项目
- [项目名]：[当前状态] — [下一步]

## 教训总结
- [时间]：[教训内容] — [如何避免]

## 技术/工具偏好
- [工具名]：[使用心得/配置]

2.5 实质内容标准

MEMORY.md 必须满足以下最低标准：

  • 总字节数 > 2000 字节（约 1000 中文字符）
  • 至少 3 个板块有实质内容

2.6 季度整理 Checklist

每季度大整理时检查：

  • 删除已完成的旧项目记录
  • 合并重复条目
  • 验证关键决策是否仍有效
  • 确认身份背景信息准确
  • 清理过时内容

三、每日 Memory（memory/YYYY-MM-DD.md）规范

3.1 命名规则

  • 标准格式：memory/YYYY-MM-DD.md
  • 多 session：同一天多次 session 在同一文件中用 --- 分隔或 ## Session N 标注，不新增文件
  • 禁止：YYYY-MM-DD-HHMM.md 时间后缀格式（容易碎片化）

3.2 内容标准

每条每日记录应包含：

# YYYY-MM-DD

## 今日完成
- [事项1]
- [事项2]

## 关键决策
- [决策内容]

## 待跟进
- [待办1]
- [待办2]

## 教训/反思（可选）
- [如果有]

3.3 禁止事项

  • 不写入完整对话内容，只写摘要
  • 不写入工具执行的完整输出（只写结论）
  • 不写入临时调试信息
  • 单文件不超过 500 行；接近时拆分为 memory/topics/ 子主题文件

四、Self-Improving 系统规范

4.1 目录位置

正确路径：

~/.openclaw/workspace-<agentId>/self-improving/

错误路径（立即迁移）：

  • ~/self-improving/ → 迁移到 workspace 内
  • ~/.self-improving/ → 迁移到 workspace 内

4.2 必需文件

文件用途更新频率
memory.md主要记忆文件随时
corrections.md纠错记录（犯过的错误）每次纠正后
reflections.md反思记录定期
heartbeat-state.json心跳状态跟踪心跳时
index.md文件索引有新增时

4.3 激活要求

  • 所有成员在 10 天内（新成员可申请延长至 14 天，需提前告知 HRBP）完成 Self-Improving 系统的实质性配置
  • 实质性配置最低标准：至少 3 个文件有实质内容（非空）
  • 禁止：报告"Setup 完成"但文件为空
  • 验证：完成后必须验证所有文件有实质内容

4.4 运作机制

  • Passive 模式（默认）：从明确纠正中学习
  • Active 模式（可选）：主动发现重复问题，同一类型错误出现 3 次以上自动建议优化

4.5 Passive 模式最低要求

每月至少一次主动回顾 corrections.md 和 reflections.md，确保学习闭环。

五、HEARTBEAT 记忆维护规范

5.1 最小配置

每个成员的 HEARTBEAT.md 必须包含以下定期任务：

# HEARTBEAT.md

## 定期 Memory 检查（约每 7-10 天一次）
- [ ] 检查 MEMORY.md 是否有待更新内容
- [ ] 从 memory/YYYY-MM-DD.md 提炼要点到 MEMORY.md（如有）
- [ ] 清理过期的待办或临时文件

注：检查频率为弹性指导，非严格计时。偶尔超过 10 天不算违规。

5.2 状态记录

在 self-improving/heartbeat-state.json 中记录：

{
  "lastMemoryCheck": 1709270400,
  "memoryEntriesCount": N
}

六、文件归档与清理规范

6.1 归档规则

  • 按项目状态判断：当某月日志所在月份已过，且该月无进行中项目的实时记录时，可归档
  • 超过 3 个月的每日日志移入 memory/archive/ 子目录
  • 已完成项目的相关文件移入 memory/topics/[项目名]/
  • 临时文件（调试、测试）使用后立即删除
  • 每个归档文件建议注明归档原因

6.2 Git 提交

每日至少提交一次，涉及重要决策的更新立即提交：

cd ~/.openclaw/workspace-<agentId>
git add memory/ MEMORY.md self-improving/
git commit -m "memory: YYYY-MM-DD 更新"

七、成员 workspace 路径说明

成员Agent IDworkspace 路径备注
诸葛zhuge~/.openclaw/workspace/（无后缀）path override 配置，与既有环境保持一致
其他成员—~/.openclaw/workspace-<agentId>/标准路径

说明：workspace 路径是技术配置，不影响规范对所有成员的同等适用性。

八、检查与问责

8.1 HRBP 检查清单

每两周一次的例行检查：

  • MEMORY.md 是否有实质内容（>2000 字节，至少 3 个板块）
  • 过去 14 天是否有每日记录
  • Self-Improving 目录是否存在且至少 3 个文件非空
  • 命名是否符合规范（无 HHMM 后缀）
  • Git 是否有定期提交

8.2 违规处理

  1. 第一次：HRBP 通知，提醒规范，给予 7 天整改期
  2. 第二次：HRBP 安排 30 分钟规范培训/答疑
  3. 第三次：上报 CEO 决策

注：违规处理是帮助成员建立习惯，不是惩罚。问责的目的是确保 Memory 系统真正发挥作用。

九、规范化执行计划

阶段时间内容
第一阶段今天各成员立即填充 MEMORY.md 身份背景和关键决策
第二阶段3 天内修复 Self-Improving 路径错误（从 ~/ 迁入 workspace）
第三阶段10 天内所有成员完成 Self-Improving 系统实质性配置（至少 3 个文件非空）
第四阶段14 天内HRBP 完成第一轮检查，确认所有成员合规
第五阶段每两周例行检查 + 持续改进

十一、免责与弹性说明

本规范是指导意见，鼓励各成员根据自身工作节奏做合理调整。HEARTBEAT 检查频率为弹性指导，非严格计时。检查与问责的目的不是惩罚，而是确保 Memory 系统真正发挥作用。
